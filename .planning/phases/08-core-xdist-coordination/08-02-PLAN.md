**Modify `docker_network` fixture:**

The current fixture unconditionally creates a Docker network. Modify to add xdist branching:

```python
@pytest.fixture(scope="session")
def docker_network(docker_network_name: str) -> Iterator[str]:
    worker_id = get_worker_id()

    # === gw1+ path: read from state, no Docker creation ===
    if not is_controller(worker_id):
        yield docker_network_name  # already resolved by docker_network_name fixture
        return

    # === gw0 / master path: create Docker infrastructure ===
    # FileLock guards against race between concurrent sessions
    if worker_id == "gw0":
        # Running under xdist — coordinate via FileLock
        if not acquire_infra_lock():
            # Another gw0 won the race (edge case: should not happen, but defensive)
            yield docker_network_name
            return
        try:
            network = _create_and_register_network(docker_network_name)
            # Write to shared state so gw1+ can proceed
            write_state_file("docker_network", docker_network_name)
        except Exception:
            # gw0 failed — signal error to gw1+ (COORD-04)
            write_state_file("error", f"Docker network creation failed: {docker_network_name}")
            raise
    else:
        # master (no xdist) — existing behavior, no state file needed
        network = _create_and_register_network(docker_network_name)

    try:
        yield docker_network_name
    finally:
        if worker_id == "gw0":
            _teardown_network(network, docker_network_name)
            release_infra_lock()
        else:
            # master teardown — existing behavior
            _teardown_network(network, docker_network_name)
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| gw1+ → State File | gw1+ trusts the state file written by gw0. A corrupted state file could cause gw1+ to use incorrect network names. |
| gw0 → FileLock | gw0 acquires FileLock to claim infrastructure ownership. Stale lock from crashed gw0 must not block restart. |
| Fixture Teardown | Docker network cleanup in gw0 teardown — if gw1+ still uses network when gw0 tears down, tests fail. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-07 | Tampering | State file JSON — gw1+ reads docker_network name | accept | State file is in `/tmp/samstack-{uuid}/` — UUID directory name prevents casual cross-session tampering. Same-host attacker with filesystem access could inject a fake network name, but at that level they already control the test environment. Testing tool, not production. |
| T-08-08 | Denial of Service | FileLock — stale lock after gw0 crash | mitigate | `release_infra_lock()` in fixture teardown finally block ensures cleanup on normal exit. On SIGKILL: fcntl (Unix) auto-releases lock when process dies; Windows requires eventual manual cleanup. The `timeout=120` default on `wait_for_state_key()` prevents gw1+ from hanging indefinitely if lock persists. Phase 12 documentation will note Windows stale-lock mitigation. |
| T-08-09 | Denial of Service | gw0 crash mid-network-creation | mitigate | `write_state_file("error", ...)` in except block per COORD-04 — gw1+ detects error and calls `pytest.skip()` within 120s timeout. No infinite hang. |
| T-08-10 | Information Disclosure | State file contains Docker network name | accept | Network name is `samstack-{uuid8}` — no secrets. Not externally routable. Access limited to same-host processes with filesystem access. |
| T-08-11 | Elevation of Privilege | FileLock path in temp directory | accept | Lock file path is `{tempdir}/samstack-{uuid}/infra.lock` — created with default permissions. No privilege escalation vector. |
</threat_model>
