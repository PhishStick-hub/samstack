<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Worker → Shared State File | Bucket names written/read via `write_state_file`/`wait_for_state_key`. No secrets — only S3 bucket names. State dir uses session UUID in temp dir, predictable only to same-session workers. |
| Test Code → Spy S3 Bucket | S3 operations on LocalStack inside Docker network. Isolated. |
| Lambda Container → Spy S3 Bucket | `spy_handler` writes spy events via `MOCK_SPY_BUCKET` env var. Lambda runs inside SAM-spawned containers on docker_network. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Spoofing | State File | accept | State file contains only bucket names, no secrets. Temp dir with session UUID limits exposure to same-host processes. |
| T-11-02 | Tampering | State File | accept | FileLock (`write_state_file` uses `_state_lock`) prevents concurrent writes. No authentication between workers — same-session trust model. |
| T-11-03 | Information Disclosure | State File | accept | Bucket names are not sensitive. All workers in same xdist session share infrastructure by design. |
| T-11-04 | Denial of Service | Spy S3 Bucket | accept | Test infrastructure only. A malicious test could flood the bucket but this is expected behavior (tests control their own data). |
| T-11-05 | Elevation of Privilege | S3 Operations | accept | All workers share the same LocalStack instance with equal access. No privilege separation between workers. |
</threat_model>
